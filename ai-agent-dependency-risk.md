## Key Findings

### 1. jsonref is a shared CRITICAL risk

Both CrewAI and AutoGen depend on `jsonref` (risk: 0.78, CRITICAL). This package
has no maintainer activity, no recent releases, and no source repository linked
in PyPI metadata. It's effectively abandoned.

**Impact:** JSON reference resolution. Both frameworks use it for JSON Schema
`$ref` handling. If a security vulnerability is found in jsonref, neither
framework can get a patch from upstream.

**Recommendation:** Fork or vendor jsonref, or migrate to an actively maintained
alternative (e.g., `referencing` from the JSON Schema team).

### 2. CrewAI has the largest attack surface

29 direct dependencies — nearly 4x LangChain Core. Notable risks:

- `tomli-w` (0.68, HIGH): No repo, no recent activity. TOML writer.
- `openpyxl` (0.64, HIGH): No repo, no recent activity. Excel file handling.
- `appdirs` (0.43): Last push 1,140 days ago. Application directory detection.
- `json5` (0.42): Single maintainer (bus factor 1).
- `pdfplumber` (0.35): Single maintainer (bus factor 1).

Two bus-factor-1 packages means two people whose departure would orphan
dependencies that CrewAI relies on.

### 3. LangChain Core is the leanest

8 direct dependencies, none above MODERATE. Worst signals:

- `pyyaml` (0.46): No commits in 6 months, slow issue response.
- `jsonpatch` (0.44): Last release 1,014 days ago.

Both are stable, mature packages — the risk is abandonment, not instability.
LangChain's tight dependency footprint is a genuine engineering advantage.

### 4. Common weak signals across all three

- **Issue responsiveness**: Most GitHub-hosted dependencies have slow first
  response times (1,000-6,000 hours median). This is normal for maintainer-run
  OSS but means security reports may go unaddressed for weeks.

- **Low close rates**: Many popular packages have <40% issue close rates.
  This doesn't indicate abandonment — it indicates triage debt. Open issues
  accumulate faster than volunteers can close them.

- **Missing metadata**: Several packages lack PyPI development status
  classifiers, security policies, or funding information. This makes
  automated risk assessment harder but doesn't indicate actual risk.

### 5. MCP SDK has a bus factor problem

The MCP Python SDK (used by all MCP servers including ours) depends on 3
single-maintainer packages:

- `typing-inspection` (0.42): Viicos is the sole maintainer
- `sse-starlette` (0.29): sysid is the sole maintainer
- `jsonschema` (0.26): Julian is the sole maintainer

None of these are abandoned — they're actively maintained by talented
individuals. The risk is personnel, not code quality. If any of these
maintainers steps away, the MCP ecosystem has a supply chain gap.

### 6. Semantic Kernel depends on an archived repo

Microsoft's Semantic Kernel depends on `nest-asyncio` (CRITICAL, 0.80) — its
GitHub repository is **archived**. The maintainer has explicitly stopped
development. Also depends on `pybars4` (HIGH, 0.53 — inactive 5 years) and
`defusedxml` (HIGH, 0.59 — inactive 569 days).

### 7. DSPy's diskcache is going stale

Stanford's DSPy depends on `diskcache` (HIGH, 0.60 — inactive 593 days).
Not abandoned yet, but heading that direction.

### 8. Haystack is moderate risk across the board

Haystack has 19 dependencies, none CRITICAL or HIGH, but 11 MODERATE.
Wide but shallow risk — many packages with minor concerns, none acute.

### 9. A2A SDK and OpenAI Agents are cleanest

Both have small dependency footprints (5 and 7) with no risks above MODERATE.
The OpenAI Agents SDK depends on `mcp` (the MCP SDK itself), creating an
interesting dependency chain: OpenAI Agents → MCP → sse-starlette (bus factor 1).

through transitive dependencies.

## Why This Matters Now

In March 2026, TeamPCP compromised LiteLLM (97M monthly downloads) through a
multi-stage supply chain attack. Over 600 projects had unpinned LiteLLM
dependencies. The attack harvested API keys for OpenAI, Anthropic, Google,
and Azure from compromised installations.

This analysis shows that similar risks exist in every major AI agent framework.
CrewAI's `jsonref` (CRITICAL, 0.78) is abandoned with no one to patch
vulnerabilities. Its `openpyxl` and `tomli-w` dependencies have no linked
source repositories — making it impossible to audit their security posture.

The question isn't whether AI agent supply chains will be targeted again.
It's whether frameworks are monitoring their dependency health before the
next incident.

## Methodology

Each package was analyzed using vigil's 4-analyzer pipeline:

1. **Maintainer**: push recency, commit trends, bus factor, release cadence
2. **Community**: issue responsiveness, close rate, contributor breadth
3. **Security**: security policy, license, yank rate, development status
4. **Sustainability**: org backing, funding, maturity, maintenance load

Signals are weighted by confidence. Risk score = 1 - (weighted health / total weight).
Stale positive signals (e.g., "Production/Stable" on abandoned packages) have
reduced confidence when last release > 2 years.

Thresholds: LOW < 0.25, MODERATE < 0.50, HIGH < 0.75, CRITICAL >= 0.75.
Validated against 20 real packages with 100% exact match accuracy.
